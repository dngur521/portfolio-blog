## Architecture

**Storage is deliberately split across two systems**, and this split drives most of the code structure:
- Post *content* (raw Markdown + frontmatter) lives in `posts/{category_slug}/{post_slug}.md` on disk.
- Post *metadata* (title, category, tags, search text, timestamps) lives in MariaDB.
- Uploaded images live under `uploads/images/{yyyy}/{mm}/{uuid}.{ext}`; only their path/metadata is in DB.

`server/services/postService.js` is the only place that writes posts, and it has to keep the filesystem and DB in sync. The create/update flow is: write the `.md` file to a temp path → open a DB transaction → `rename()` the temp file over the real path *before* `commit()` → commit. This ordering (rename before commit, not after) is intentional: if the rename fails, the transaction can still be rolled back; if the commit fails after a successful rename, the code deletes/restores the file as a compensating action. When updating a post in place (same category+slug), a `.bak-*` copy of the original file is made first so a failed commit can restore it. Read this function before changing post create/update logic — it's easy to reintroduce a "file exists but DB doesn't" (or vice versa) bug by reordering steps.

**Request flow**: `routes/*.js` → `services/*.js` → either `db/pool.js` (mysql2 promise pool, prepared statements only) or the filesystem directly. Routes do not talk to the DB or filesystem themselves. `server/middlewares/auth.js` re-checks `is_active` in the DB on *every* admin request (not just at login) so that deactivating an account invalidates existing sessions immediately.

**Sessions & CSRF**: sessions are stored in MariaDB via `express-mysql-session`. `csurf` is mounted globally on `/api` in `server/app.js`; the frontend must first `GET /api/csrf-token` and send the value back as an `X-CSRF-Token` header on any mutating request (see `public/js/common.js`'s `fetchJSON` wrapper, which does this automatically). The session cookie has `secure: true` in production, which means it is **silently dropped** unless the request looks HTTPS to Express — see the trust-proxy note below.

**Reverse proxy / HTTPS**: this app runs as a plain HTTP server behind a Cloudflare Tunnel (`cloudflared`, running as a systemd service on the host, ingress configured in the Cloudflare Zero Trust dashboard — not in any local `config.yml`). `app.set('trust proxy', 1)` in `server/app.js` assumes exactly one hop (cloudflared) sets `X-Forwarded-Proto`. When testing locally with `curl` directly against `localhost:3000`, you must add `-H "X-Forwarded-Proto: https"` or the session cookie will not be set and every authenticated request will look logged-out.

**Upload validation**: `server/services/uploadService.js` uses `file-type` (v17+, ESM-only) via a dynamic `import()` from this otherwise-CommonJS codebase — don't `require()` it. Uploaded bytes are validated by binary signature (not by client-provided MIME type or extension), then re-encoded through `sharp` to strip EXIF, except animated GIFs which are passed through unmodified to preserve frames.

**Slugs / path safety**: `server/utils/sanitizeSlug.js` provides `assertValidSlug` (enforces `^[a-z0-9-]+$`) and `safeResolve` (resolves a path and verifies it stays inside the intended root dir). Any code building a filesystem path from a category/post slug or upload filename must go through these — this is the app's main defense against path traversal.

**About page**: unlike posts, the "About me" content (`server/services/aboutService.js`, `GET /api/about`, `PUT /api/admin/about`) has no DB row at all — it's a single Markdown file at `content/about.md`, written via a plain temp-file-then-`rename` (no DB transaction needed since there's no metadata to keep in sync). Don't force this into the `posts` table/category system; it's intentionally simpler because it has no title/slug/tags/search-indexing needs.

**Frontend nav/auth model**: there is exactly one nav renderer, `Blog.renderNav(activePage)` in `public/js/common.js`, used by *every* page (public and admin). It fetches only `/api/auth/status` and renders the same public-looking nav everywhere; if authenticated it swaps the "관리자 로그인" button for logout + 이력조회/계정관리 links and returns the status object. The nav does **not** list categories anymore (removed — it duplicated the per-page "카테고리별 보기" dropdown and broke the nav's layout); category browsing lives entirely in `Blog.renderCategoryDropdown()`. There is no separate always-on editor UI: `/admin/editor` and `/admin/about` render a blank/prefilled Toast UI Editor only when navigated to directly (via `?category=&slug=` query params for a specific post, or with no params for a new one), reached only through "새 글 작성"/"수정" buttons that the public pages (`/`, `/category`, `/post`, `/about`) show *only when `renderNav()`'s returned status is authenticated*. Admin-only pages with no public equivalent (`/admin/logs`, `/admin/accounts`, and the two editor pages) call `Blog.redirectIfNotAuthenticated(status)` right after `renderNav()` to bounce anonymous visitors to `/admin/login`. Don't reintroduce a separate "admin mode" landing page that shows the editor by default — that was deliberately removed in favor of this browse-then-edit-on-click model.

**Categories**: `GET /api/categories` hides categories with zero posts by default (`categoryService.listCategories({ includeEmpty })`) so visitors never see a link into an empty category. `public/admin/editor.html`'s category `<select>` needs *every* category including empty ones (you have to be able to pick a category you just created before it has any posts), so `admin-editor.js` is the one caller that passes `?includeEmpty=1`. Creating a category from the editor uses a plain inline form (`#new-category-form`, shown/hidden with jQuery) — **not** `window.prompt()`; that was tried first and the user rejected it as unusable when the dropdown has nothing but "+ 새 카테고리 만들기" in it (see [[feedback_frontend_conventions]]). Category deletion is a button next to the category `<select>` on the same editor page, calling the existing `DELETE /api/admin/categories/:slug` (still blocked server-side with 409 while the category has posts). Since the FK constraint already guarantees the category has zero posts by the time the DB row is actually deleted, `categoryService.deleteCategoryBySlug` also `fs.rm(categoryDir, { recursive: true, force: true })`s the now-empty `posts/{slug}/` directory right after — otherwise the empty folder was left behind on disk forever.

**Search**: `server/services/searchService.js` uses `LIKE '%query%'` against `title`/`search_text` (title matches ranked above body-only matches, tie-broken by `published_at DESC`), not MariaDB `FULLTEXT ... NATURAL LANGUAGE MODE`. FULLTEXT was tried first and silently failed on short Korean queries — InnoDB's default minimum token size (3 chars) combined with the built-in parser's lack of Korean word segmentation meant a 2-character query like "테스" could never match "테스트1", regardless of any app-level minimum-length check. `LIKE` does plain substring matching so it works regardless of token/word boundaries; user input is escaped for `%`/`_`/`\` before being wrapped in `%...%` (see `escapeLikePattern`) since it's concatenated into the pattern string, not passed as a separate parameter to `LIKE`. `server/routes/search.js` only rejects an empty query now (no more arbitrary 2-char minimum).

**Self-hosted frontend libraries**: jQuery, `marked`, `DOMPurify`, `highlight.js`, and `mermaid` are installed as regular npm dependencies and served from `node_modules` as-is (no bundler) via `/vendor/*` static mounts in `server/app.js` (e.g. `/vendor/jquery/jquery.min.js` → `node_modules/jquery/dist/jquery.min.js`), instead of loading them from a CDN. `highlight.js` specifically uses the separate `@highlightjs/cdn-assets` package, not the plain `highlight.js` package — the latter's npm build is CommonJS-only with no browser-ready bundle (its `lib/index.js` uses `require()`, which doesn't run as a plain `<script>`), while `@highlightjs/cdn-assets` is exactly the prebuilt browser bundle highlight.js publishes for this use case. `mermaid`'s npm package ships `dist/mermaid.min.js`, a genuinely self-contained IIFE bundle (bundles its own DOMPurify/js-yaml/lodash-es/etc. and assigns `globalThis["mermaid"]`) unlike Toast UI Editor below, so it self-hosts cleanly via `/vendor/mermaid`. **Toast UI Editor is the one exception and stays on the `uicdn.toast.com` CDN**: its npm package's `dist/toastui-editor.js` externalizes `prosemirror-*` as UMD "externals" meant to be resolved by a bundler, and literally does `root[undefined]` in the plain-`<script>` fallback branch of its UMD wrapper — it cannot work as a bare `<script>` tag under any circumstances without a bundler. The self-contained `toastui-editor-all.min.js` bundle that CDN use requires only exists on NHN's CDN, not in the npm-published package. If you're tempted to self-host Toast UI Editor too, don't — it was tried and reverted (`Uncaught TypeError: Cannot read properties of undefined (reading 'PluginKey')`) for exactly this reason. `server/app.js`'s CSP `scriptSrc`/`styleSrc` reflects this split: `'self'` (for `/vendor/*`) plus `https://uicdn.toast.com` (Toast UI only) plus `https://static.cloudflareinsights.com` (Cloudflare's own auto-injected Web Analytics beacon, not app code — `connectSrc` also allows `https://cloudflareinsights.com` for the beacon's own reporting calls).

**Markdown rendering & Mermaid diagrams**: `Blog.renderMarkdown(rawMarkdown)` in `common.js` is the one place that turns post/About Markdown into sanitized HTML (used by `post-viewer.js` and `about.js` — don't call `marked.parse`/`DOMPurify.sanitize` directly elsewhere). It installs a custom `marked.Renderer#code` that intercepts ` ```mermaid ` fenced blocks and emits `<pre class="mermaid">` (escaped raw source) instead of the default `<pre><code class="language-mermaid">`, so mermaid can find and replace them while regular code blocks still go through `highlight.js` as before (mermaid blocks have no `<code>` child, so `.post-body pre code` never matches them). `Blog.renderMermaidDiagrams(container)` must be called after the HTML is inserted into the DOM — it lazily calls `mermaid.initialize({ theme })` once (light/dark picked from the current `data-theme`/`prefers-color-scheme` at call time, not re-evaluated on later theme toggles) and then `mermaid.run({ nodes })` on that container's `.mermaid` elements. No CSP changes were needed for this — `mermaid.min.js` is loaded from `/vendor` (`'self'`) and its one `Function("return this")()` global-detection fallback is unreachable in a browser (short-circuited by `globalThis`/`self` already existing), so it never actually needs `'unsafe-eval'`.

**Pagination**: `Blog.renderPagination(elId, { page, totalPages, onChange })` in `common.js` is a shared prev/next+page-indicator UI, reused by both server-driven (이력 조회) and client-driven (전체 글) pagination; it renders nothing when `totalPages <= 1` rather than showing a dead "1 / 1" control, since that used to read as "there's no pagination" rather than "there's only one page". `index.html`'s 전체 글 목록 paginates client-side over the already-cached `cachedPosts` array (`dashboard.js`, page-size `<select id="page-size">` with 10/25/50 options, resets to page 1 on sort or page-size change) — no extra fetch. `/admin/logs` paginates server-side as before (`activityLogService.listLogs`'s existing `page`/`limit` params), just with the page size now driven by a `<select id="log-page-size">` (10/25/50, default 50) instead of a hardcoded `limit=50`.

**Admin git commit/push**: `/admin/git` (`public/admin/git.html` + `public/js/admin-git.js`, linked from the nav next to 이력 조회/계정 관리) is a dedicated page — not an inline panel in the editor, that was tried first and the user couldn't find it since it only appeared for a few seconds right after saving a post. It lists every uncommitted change under `posts/` and `content/about.md` (`GET /api/admin/git/status` → `gitService.getStatus`, parsed `git status --porcelain` output, status one of `new`/`modified`/`deleted`) with a checkbox per item (all checked by default), a "메시지 추천받기" button, and a "선택 항목 커밋 & 푸시" button — never automatic, and the admin can select any subset of the listed files to commit together as one commit. `server/services/geminiService.js` calls the Gemini API (`GEMINI_API_KEY`/`GEMINI_MODEL` in `.env`, not committed — see `.env.example`) with the selected files' current content (or "삭제됨" for deleted ones), asking for one line in this repo's `<emoji> <type>: <summary>` convention covering all of them. `server/services/gitService.js` runs `git add`/`git commit -m ... -- <paths>`/`git push` via `execFile` with argument arrays (never a shell string), scoped to exactly the selected paths so it never sweeps up unrelated working-tree changes. `server/routes/adminGit.js` exposes `GET /status`, `POST /suggest-message`, and `POST /commit-push` (all under `/api/admin/git`, behind `requireAuth`); every path from the client is validated against `^posts/[^/]+/[^/]+\.md$` (via `assertValidSlug`/`safeResolve`) or must equal `content/about.md` *exactly* before it's allowed anywhere near a `git` command — this is the feature's only defense against an arbitrary path reaching `execFile('git', ...)`. This requires the deploy host to actually have a working git push credential for `origin` (this repo uses `credential.helper store` + a GitHub PAT seeded into `~/.git-credentials`, mode 600, for the `kam` OS user that both the interactive shell and the systemd service run as) — don't assume this exists on a fresh host without checking.

**Post list sorting & caching**: `index.html`/`category.html` fetch the post list *once* per page load and keep it in a module-level `cachedPosts` array (`dashboard.js`/`category.js`); the 이름순/날짜순/수정순 × 오름차순/내림차순 controls (`Blog.sortPosts()`, `Blog.initSortControls()` in `common.js`) just re-sort and re-render that array client-side — changing sort never re-fetches. Default is `date`/`desc`, matching the pre-existing `ORDER BY published_at DESC` behavior. `수정순` needs `updatedAt` on each list item, so `postService.listPosts()` selects `p.updated_at` even though earlier versions only needed it on the detail endpoint.

**Perf**: page-init scripts fire their independent API calls (`renderNav()`'s auth check, category dropdown, post list, etc.) via `Promise.all` rather than sequential `await`s — these calls don't depend on each other, so awaiting them one at a time was pure wasted latency. `express.static` in `server/app.js` also sets `Cache-Control: public, max-age=86400` on `.css`/`.js` (not on `.html`, which stays uncached since it's still actively changing) so navigating between pages doesn't even do a conditional-GET round trip for shared assets. This app is a classic multi-page site (no client-side router), so a full document load still happens on every link click — that's inherent to the architecture, not something either of these two changes fixes, and turning it into an SPA was explicitly decided against as overkill for this project's size.

**Activity log**: `activity_logs` table (renamed from `auth_logs` — live DB was migrated with `ALTER TABLE ... RENAME TABLE`, not dropped) now covers auth events (`LOGIN_SUCCESS`/`LOGIN_FAIL`/`LOGOUT`), content events (`POST_CREATE`/`POST_UPDATE`/`POST_DELETE`), and `GIT_COMMIT_PUSH`, via `server/services/activityLogService.js` (renamed from `authLogService.js`). `event_type` is a real SQL `ENUM` (see `server/db/schema.sql`), so adding a new event type needs both an `ALLOWED_EVENTS` update *and* a live `ALTER TABLE ... MODIFY COLUMN event_type ENUM(...)` on the deployed DB — the app's own `blog_app` DB user can't run `ALTER TABLE` (only `SELECT/INSERT/UPDATE/DELETE`), so this has to be run manually as DB root, same as the `auth_logs`→`activity_logs` rename was. Auth events still get logged from inside `authService.js`/`rateLimiter.js`. Post events are logged from `server/routes/admin.js` (`logActivity()` helper); the git commit/push event is logged the same way from `server/routes/adminGit.js` (its own identical local `logActivity()` helper, not shared — see [[project_admin_git_commit_feature]]) — neither `postService.js` nor `gitService.js` take a dependency on `req`, so the route layer is where the logging call has to live. Every `logActivity()` swallows its own errors (logs to console, doesn't `next(err)`) so a logging failure can never turn a successful action into an error response to the client. The `target` column holds a human-readable string (`"{category}/{slug} - {title}"` for posts, `"{comma-separated paths} - {commit message}"` for git); deleting a post fetches its detail *before* calling `postService.deletePost()` specifically so the log can still record its title afterward. The admin UI page is titled "이력 조회" (not "로그인 이력" anymore) at the same `/admin/logs` URL.

**Clean URLs**: every page file on disk still ends in `.html` (e.g. `public/about.html`), but nothing should ever *link* to a `.html` path. `server/app.js` has a redirect (`app.get(/\.html$/, ...)`, 301 to the extensionless path, `/index.html` → `/`) placed before `express.static(..., { extensions: ['html'] })`, which is what makes `/about` resolve to `about.html`. When adding a new page, give it a normal `.html` filename as usual, but write every internal `href`/`window.location.href` to it without the extension.

**Dark mode**: theme is a `light` / `dark` / `auto` choice stored in `localStorage['theme']` ('auto' = no override, follows `prefers-color-scheme`). Every page's `<head>` has the *same* inline script (byte-for-byte identical on purpose) that applies a saved `light`/`dark` choice to `<html data-theme>` before first paint, to avoid a flash of the wrong theme; `public/css/style.css` defines the actual colors three times — base `:root` (light default), `@media (prefers-color-scheme: dark)`, and `:root[data-theme='dark']`/`:root[data-theme='light']` (explicit choice overrides the media query). Because CSP has no `'unsafe-inline'` for scripts, that inline snippet is allowlisted in `server/app.js`'s `scriptSrc` by its exact sha256 hash — **if you ever edit the inline script's text, you must recompute this hash** (e.g. `crypto.createHash('sha256').update(exactScriptText).digest('base64')`) and update it in every HTML file *and* `server/app.js`, or the theme script gets silently blocked by the browser. `Blog.renderNav()` renders the light/dark/auto `<select>`; pages without a nav (`admin/login.html`) call `Blog.initThemeToggle()` directly on their own standalone `<select>`.

**Deployment**: runs as the `portfolio-blog` systemd service (unit file checked into `deploy/portfolio-blog.service`, installed to `/etc/systemd/system/`), as the OS user `kam` (the Linux account on the host — unrelated to the deleted blog admin account of the same name). **Binds to `127.0.0.1` only** (`app.listen(port, '127.0.0.1', ...)` in `server/app.js`) — this was originally omitted and bound to all interfaces, which let LAN/external traffic reach the app directly bypassing Cloudflare; don't remove the explicit host argument. Public hostname routing is `blog.doomfan.win` → `localhost:3000` via the Cloudflare Tunnel named `nextcloud` (dashboard-managed, see the tunnel's Public Hostname config in the Cloudflare Zero Trust dashboard — not in any file on disk).
